# MCP 迁移方案纪要

更新时间：2026-05-08

## 1. 当前结论

- 当前项目里原来叫 `mcp` 的部分，本质是本地工具抽象。
- 它不是 Spring AI 官方 MCP Client/Server 协议接入。
- 当前工具能力主要依赖本地 Java Bean、`@Tool`、注册表和执行器。

## 2. 为什么不是官方 MCP

- 当前工具调用是 JVM 内本地执行。
- 工具不是通过标准协议暴露给模型。
- 没有远端 MCP Server 与 MCP Client 的标准发现、连接、调用过程。

## 3. 官方 MCP 更适合的场景

- 跨服务工具调用
- 跨语言工具接入
- 独立部署的工具服务
- 运行期动态发现工具

## 4. 当前项目建议

- 短期不要全量切官方 MCP。
- 先稳定业务能力边界与输入输出 schema。
- 再按服务边界逐步迁移。

## 5. 推荐优先迁移对象

- 房源查询能力
- 海报生成能力
- 营销内容资产能力
- 渠道发布能力

## 6. 引入官方 MCP 后的影响

- 不会推翻“模型直调工具”和“Planner + 执行器”两条链路。
- 主要变化在工具接入层：
  - 工具来源从本地 Bean 变为远端 MCP Server
  - 执行器需要支持本地调用与 MCP 调用双通道

## 7. 推荐迁移路线

1. 先稳定本地 Tool 的能力边界
2. 再引入 MCP Client 代理层
3. 逐步把适合独立部署的能力拆为 MCP Server
4. 让 Planner 和执行器兼容本地 Tool 与远端 MCP Tool
