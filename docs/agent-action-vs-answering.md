# Agent 服务：回答型与行动型的区别

## 背景

目前这套 `agent-service` 已经不是单纯的聊天问答服务。它具备：

- 基于 `DeepSeek` 的大模型推理能力
- 基于 `MCP client/server` 的跨服务工具调用能力
- 基于 `Planner` 的分步执行能力
- 基于 `Dubbo` 的正常业务 RPC 能力

但从当前能力结构看，它更偏向“回答型 Agent”，也就是：

- 帮用户理解问题
- 帮用户检索知识
- 帮用户查询业务数据
- 帮用户做分析、总结和建议

如果要让 Agent 真正“帮用户做一件事”，而不是只返回一个总结答案，那么系统重点就要从“回答链路”切换到“执行链路”。

## 当前方案更像什么

当前 `agent-service` 的两种主要执行模式是：

### 1. TOOL 模式

链路通常是：

1. 用户输入问题
2. 大模型判断是否需要调用工具
3. 通过 `MCP tool` 或本地工具获取结果
4. 大模型基于工具结果组织最终回答

这种模式适合：

- 知识问答
- 房源检索
- 对比分析
- 报告生成
- 营销内容草稿生成

它的核心产出是：`answer`

### 2. PLANNER 模式

链路通常是：

1. 用户输入一个相对复杂的目标
2. Planner 先拆解执行步骤
3. 每一步调用对应工具
4. 聚合步骤结果
5. 返回最终总结

这种模式比 TOOL 模式更强，但目前依然偏向“为了得到答案而分步执行”，不是“为了真正完成业务动作而分步执行”。

它的核心产出依然主要是：`answer`

## 如果要变成“行动型 Agent”，区别在哪里

当用户说：

- “帮我把这套房发布到 3 个平台”
- “帮我给下周到期委托客户发提醒”
- “帮我生成对比报告并推送给店长”
- “帮我批量创建本月结算批次并标记待打款”

这类需求就不再是“回答问题”，而是“执行任务”。

这时 Agent 的目标不再是给结论，而是完成动作。

## 当前回答型 Agent 与行动型 Agent 的核心差异

### 1. Tool 的性质不同

当前多数工具偏只读：

- `query`
- `search`
- `list`
- `analyze`
- `generate-report`

行动型 Agent 需要大量具备副作用的工具：

- `create`
- `update`
- `submit`
- `send`
- `publish`
- `approve`
- `trigger`

也就是说，当前多数 Tool 是“查”，行动型 Tool 是“办”。

### 2. Planner 的目标不同

当前 Planner 更像：

- 为了回答问题而拆步骤

行动型 Planner 更应该是：

- 为了完成任务而拆步骤

这要求每一步不仅要描述“做什么”，还要明确：

- 前置条件
- 入参从哪里来
- 是否允许自动执行
- 是否需要人工确认
- 是否可以重试
- 失败后是否需要补偿或回滚

### 3. 返回结果模型不同

当前回答型 Agent 更关注：

- 最终答案
- 工具结果摘要
- 简单上下文

行动型 Agent 更关注：

- 任务状态
- 步骤状态
- 创建或更新的业务对象 ID
- 外部平台回执
- 成功数量与失败数量
- 可重试步骤
- 下一步建议动作

也就是说，输出从 `answer` 变成 `task result`

### 4. 风险控制要求不同

回答型 Agent 如果判断错了，通常是“答得不准”。

行动型 Agent 如果执行错了，可能会直接造成业务事故，例如：

- 错发消息
- 错发布内容
- 错改状态
- 错创建结算批次

因此行动型 Agent 必须补齐：

- 权限校验
- 幂等控制
- 高风险确认
- 审批节点
- 操作审计
- 失败补偿

### 5. MCP Server 的能力设计不同

当前 MCP Server 更偏“能力查询接口”。

行动型 Agent 需要 MCP Server 提供“事务化动作接口”，例如：

- `create_contract`
- `publish_marketing_content`
- `send_notification`
- `create_settlement_batch`
- `update_listing_status`

也就是说，当前很多工具是数据能力，后续要补业务动作能力。

## 目前方案已经具备的基础

要演进成行动型 Agent，并不需要推翻现在的架构。

当前已经具备的底座包括：

- `DeepSeek` 大模型决策能力
- `MCP client/server` 跨服务工具调用能力
- `TOOL / PLANNER` 双模式
- Planner 会话和步骤日志
- 各业务服务已经具备一定的数据查询与业务处理能力

这些已经足以支撑下一阶段改造。

## 目前还缺少的关键能力

如果要让 Agent 真正替用户做事，当前还需要补齐以下能力：

### 1. 带副作用的 MCP Tool

不是只查数据，而是能真正执行业务动作：

- 创建
- 提交
- 发布
- 发送
- 推进状态

### 2. 任务状态机

需要区分任务执行过程，例如：

- `PENDING`
- `PLANNING`
- `WAITING_CONFIRMATION`
- `RUNNING`
- `PARTIAL_SUCCESS`
- `FAILED`
- `SUCCEEDED`

### 3. 幂等与重试机制

对于通知发送、营销发布、结算批次创建这类动作，必须避免重复执行。

### 4. 人工确认节点

有些动作不能直接自动执行，应该先让 Agent 给出计划，再由用户确认，例如：

- 批量通知
- 批量发布
- 批量结算
- 合同发起签署

### 5. 审计日志

要能追踪：

- 谁发起的任务
- Planner 生成了什么计划
- 每一步调用了什么工具
- 每一步传了什么参数
- 哪一步失败
- 最终结果是什么

### 6. 失败补偿或回滚策略

例如：

- 已生成内容但未成功发布
- 已创建批次但未成功打款
- 已发送部分通知但有一部分失败

这时系统需要明确定义补偿逻辑，而不是只返回错误。

## 对当前系统的判断

一句话概括：

当前方案是“会查、会想、会总结的 Agent”。

如果要升级成“会真正替你办事的 Agent”，不需要推翻当前 `MCP + Planner + DeepSeek + Dubbo` 架构，但要把重点从“回答链路”切到“执行链路”。

## 建议的演进路径

不建议一上来就把所有能力都做成全自动执行，建议分三层推进。

### 第一层：Read Agent

只负责：

- 查询
- 分析
- 总结
- 建议

这部分当前已经基本具备。

### 第二层：Assist Agent

可以生成计划，但先不直接执行，先给用户确认，例如：

- “已准备发布到贝壳、安居客、抖音，是否确认执行？”
- “已准备向 18 位委托即将到期客户发送提醒，是否确认？”

### 第三层：Action Agent

确认后真正执行，并返回完整执行结果，例如：

- 成功数量
- 失败数量
- 各步骤状态
- 外部系统回执
- 可重试入口

## 推荐优先落地的场景

建议先选一个真正闭环、风险可控的动作场景做试点，而不是一开始就全量自动化。

比较适合当前架构优先落地的场景包括：

### 场景 1：批量发送委托到期提醒

原因：

- 数据来源明确
- 执行动作清晰
- 风险相对可控
- 很适合 `Planner + Notification MCP tool`

### 场景 2：生成营销内容并一键发布

原因：

- 已有营销内容和推广相关服务基础
- 可以很好体现 `生成 + 审核 + 发布` 的执行链路
- 适合验证人工确认、发布结果回执、失败重试等机制

## 总结

当前 `agent-service` 并不是只能做内容回答和知识检索，但现阶段它的主能力确实仍然偏“回答型”。

如果目标是让 Agent 真正替用户做事，那么与现在方案的核心区别不在于是否换模型，也不在于是否保留 Planner，而在于：

- 工具是否具备真实执行能力
- Planner 是否围绕任务执行设计
- 系统是否具备执行治理能力

后续改造重点应该放在：

- 行动型 MCP Tool
- 任务状态机
- 人工确认
- 幂等重试
- 审计与补偿

而不是单纯继续优化回答文本。
