# 分布式多智能体记忆框架设计文档

## 概述

本记忆框架为贝壳找房中台多智能体系统提供四层记忆能力，覆盖跨 Agent 工作流历史、审批检查点版本化、用户习惯偏好学习、系统级业务约束注入四个维度。

**核心原则**：子 Agent 保持无状态——所有记忆由 supervisor 在构建 A2A 请求时注入，子 Agent 零改动。

---

## 四层记忆架构

│  └──────────────┘ └──────────────┘ └──────────────────────┘   │
│         │                │                   │                  │
│         ▼                ▼                   ▼                  │
│  ┌──────────────────────────────────────────────────────┐      │
│  │              sharedContext + A2A request               │      │
│  │  · userPreferences      · workflowHistory             │      │

### 第一层：跨 Agent 工作流记忆（Session 级）

**目标**：下游子 Agent 能看到完整的上游操作历史。

**数据来源**：
- `handoff_relation_memory` 表——Agent 间切换关系
- `task_artifact_memory` 表——各环节产出的制品记录

**读取路径**：
- `WorkflowHistoryService.assembleHistory(taskId)` 从已有表组装时间线
- `BuildNextAgentContextNode.build()` 注入 `workflowHistory` 和 `upstreamArtifactSummaries` 到 context

**生命周期**：Session/Task 级，跟随工作流生命周期。

---

### 第二层：审批 Checkpoint 记忆（Task 级）

**目标**：支持单个工作流中多次审批，不会互相覆盖。

**核心改动**——`agent_workflow_checkpoint` 表：

| 字段 | 说明 |
|------|------|
| `task_id` | 工作流任务 ID |
| `checkpoint_version` | 版本号，每次 save 自增 |
| `snapshot_json` | 完整状态快照 |

唯一键从 `(task_id)` 改为 `(task_id, checkpoint_version)`。

**行为变化**：

| 操作 | 旧行为 | 新行为 |
|------|--------|--------|
| `save()` | upsert（同 taskId 覆盖） | 始终 insert 新行，version +1 |
| `load()` | 查任意一行 | 查最新版本（ORDER BY version DESC LIMIT 1） |
| `delete()` | 删一行 | 删指定 taskId 所有版本 |

**实现位置**：
- `AgentWorkflowCheckpointEntity.java`——新增 `checkpointVersion` 字段
- `DbGraphCheckpointStore.java`——版本化读写逻辑
- `InMemoryGraphCheckpointStore.java`——内存版行为不变（键为 taskId，本质覆盖）

---

### 第三层：用户习惯记忆（User 级，长期）

**目标**：LLM 自动从对话中提取用户偏好，下次会话自动注入。

**双存储设计**：

| 存储 | 用途 | 场景 |
|------|------|------|
| MySQL `user_preference_memory` | 精确查询，confidence 管理 | 按 userId 加载所有偏好 |
| Milvus `user_preference_knowledge` | 语义模糊匹配 | 用户说"东边"匹配到 `preferred_district: 朝阳` |

**写入——UserPreferenceCollector**（异步）：

1. `PersistSessionNode.persist()` 完成后触发 `@Async` 收集
2. 构建会话摘要（用户消息 + sharedContext + summary）
3. 调用 LLM 提取偏好 JSON（confidence >= 0.7 才保存）
4. 写入 MySQL（upsert by `userId + preferenceKey`）
5. 嵌入写入 Milvus
6. 衰减未被观察到的已有偏好（confidence -0.05，低于 0.1 自动删除）

**LLM 提取 Prompt 要求**：
```json
[
  {
    "key": "preferred_district",
    "value": "朝阳区",
    "category": "location",
    "confidence": 0.9,
    "evidence": "用户三次提到只考虑朝阳区的房源"
  }
]
```

**读取——UserPreferenceRetriever**（在 LoadSessionNode 中调用）：

```
userId → MySQL 精确查询 (WHERE user_id = ? AND confidence >= 0.5)
      → Milvus 语义搜索 (userQuery，topK=3)
      → 去重合并 → sharedContext.userPreferences
```

**confidence 生命周期**：
```
初始 ≥ 0.7 → 每次未观察到 -0.05 → 低于 0.1 自动删除
```

### 第四层：系统约束记忆（系统级，永久）

**目标**：业务规则、术语表、反幻觉约束不硬编码，而是可检索的知识。

**双存储设计**：

| 存储 | 用途 |
|------|------|
| MySQL `system_constraint_memory` | 结构化存储，支持按 category/tags 查询 |
| Milvus `system_constraint_knowledge` | 嵌入语义搜索，支持 LLM tool 调用 |

**Bootstrap——SystemConstraintBootstrapRunner**：

启动时从 `system-constraints.yml` 读取 18 条约束，写入 MySQL + 嵌入写入 Milvus。

**YAML 示例**：
```yaml
constraints:
  - key: "ke_no_commercial"
    text: "贝壳找房平台仅处理住宅类房源交易，不支持商业地产。"
    category: "domain_rule"
    tags: "listing,anti_hallucination"
  - key: "commission_rate_range"
    text: "买卖佣金范围为成交价的1%至3%。"
    category: "policy"
    tags: "settlement,commission"
  - key: "vocab_daikan"
    text: "带看：指经纪人带领客户实地查看房源的过程。"
    category: "vocabulary"
    tags: "listing,agent"
```

**读取路径**：

1. **Push 路径**：`LoadSessionNode.loadRelevantConstraints(userMessage)` 根据用户消息关键词匹配领域标签，注入 `sharedContext.systemConstraints`
2. **Pull 路径**：supervisor LLM 通过 `milvusKnowledgeSearch` 工具按需检索
3. **子 Agent 注入**：`BuildInvokeRequestNode.buildConstraints()` 将约束文本注入 `AgentTaskInvokeRequest.constraints`

**约束分类**：

| category | 标签示例 | 数量 |
|----------|----------|------|
| `domain_rule` | listing, anti_hallucination, trade, settlement | 6 |
| `policy` | settlement, commission, contract | 5 |
| `vocabulary` | listing, agent, trade | 4 |
| `workflow_rule` | approval, parallel | 3 |

## 数据流全景

```
启动阶段:
  SystemConstraintBootstrapRunner
    ├── 读取 system-constraints.yml
    ├── 写入 MySQL (system_constraint_memory)
    └── 嵌入写入 Milvus (system_constraint_knowledge)

  MemoryCollectionInitializer
    ├── 初始化 Milvus collection: user_preference_knowledge
    └── 初始化 Milvus collection: system_constraint_knowledge

会话开始 (LoadSessionNode):
  ├── MemoryStoreClient.getSessionMemory(sessionId) → 恢复 sharedContext
  ├── UserPreferenceRetriever.retrieve(userId, userMessage)
  │     ├── MySQL: getUserPreferences(userId, null)
  │     └── Milvus: searchKnowledge("user_preference_knowledge", query, 3)
  └── loadRelevantConstraints(userMessage)
        └── 关键词匹配 → searchSystemConstraints(tags)

Agent 切换 (BuildNextAgentContextNode):
  └── WorkflowHistoryService.assembleHistory(taskId)
        ├── listHandoffsByTask(taskId) → workflowHistory
        └── listArtifactsByTask(taskId) → upstreamArtifactSummaries

A2A 请求构建 (BuildInvokeRequestNode):
  └── buildConstraints(sharedContext.systemConstraints) → List<String>

会话结束 (PersistSessionNode):
  ├── upsertSessionMemory() → session_shared_memory 表
  ├── saveSessionMemorySnapshot() → session_memory_snapshot 表 (版本化)
  └── UserPreferenceCollector.collectAsync()
        ├── LLM 提取偏好
        ├── upsertUserPreference() → user_preference_memory
        ├── upsertMemory() → Milvus user_preference_knowledge
        └── decayUserPreferences()
```

## 设计决策记录

1. **子 Agent 零改动**：所有记忆通过 supervisor 在 A2A 请求中注入，子 Agent 只需读取 `structuredContext` 和 `constraints` 字段，无需任何代码修改。

2. **双存储（MySQL + Milvus）**：MySQL 负责精确查询和结构化过滤（如 WHERE user_id = ? AND confidence >= 0.5），Milvus 负责语义模糊匹配（如"东边的房子"→"preferred_district: 朝阳"）。

3. **LLM 驱动 + 规则兜底**：用户偏好由 LLM 判断提取（confidence >= 0.7），系统约束启动时从 YAML bootstrap。两者都具备 confidence 衰减/active 标记的管理机制。

4. **Checkpoint 版本化**：改为始终 insert 新行而非 upsert，解决多次审批互相覆盖的问题。旧版本保留用于审计追溯。

5. **异步收集不阻塞主流程**：UserPreferenceCollector 使用 `@Async` 异步执行，LLM 调用失败不影响会话正常结束。

6. **复用现有基础设施**：复用 memory-service（MySQL）、Milvus、RestClient、ObjectMapper 等已有组件，不引入新依赖。
